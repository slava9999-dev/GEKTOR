# Контекст для консилиума ИИ-моделей: Переход Gerald-SuperBrain на облачные LLM

## 1. Описание проекта (Gerald-SuperBrain)

**Gerald-SuperBrain** — это персональный гипер-интеллектуальный ИИ-ассистент (названный "Gerald"), интегрированный с Telegram и ориентированный на автоматизацию крипто-трейдинга, системный мониторинг и выполнение локальных Python-скриптов.

Проект состоит из нескольких ключевых компонентов, работающих асинхронно на одном сервере (ОС: Windows):

1. **Ядро Агента (GeraldAgent):**
   - Получает сообщения из Telegram (`bridge_v2.py`).
   - Использует RAG (VectorDatabase на базе ChromaDB/LanceDB) для поиска контекста.
   - Имеет семантический кэш (`SemanticCache`) для быстрых ответов на повторяющиеся вопросы.
   - Строгий формат вывода: Агент обязан возвращать валидный JSON, соответствующий Pydantic-схеме `AgentOutput` (содержит поля `thought`, `tool_name`, `tool_args`).
   - Поддерживает вызов инструментов (Tool Calling): `execute_python_code`, `read_file`, `final_answer`.
2. **Крипто-скиллы (Crypto Skills):**
   - **Crypto Sentinel PRO:** Мониторинг рынка Bybit (WebSockets и REST) на предмет спуфинга, крупных ликвидаций, CVD-всплесков и плотностей в стакане.
   - **Gerald Sniper:** Поиск математических уровней поддержки/сопротивления (используя ядерную оценку плотности - KDE из `scipy`) и отправка торговых сетапов в Telegram.
3. **Хранилище:**
   - SQLite для логов, истории сессий и торговых данных.
   - Локальная база векторов для RAG.

## 2. Текущая проблема и ограничения

В данный момент ИИ-ядро использует локальный инференс через **`llama-cpp-python`** (модель `Qwen2.5-7B-Instruct`, квантованная версия).

- **Критический недостаток:** Время ответа агента составляет от **80 до 95 секунд** (`SLA Warning: Response took 92.44s`). Это неприемлемо для интерактивного Telegram-бота.
- **Ограничение ресурсов:** Запуск тяжелой LLM выедает VRAM (ограничение в 7.8 ГБ) и ресурсы CPU, что мешает стабильной работе `Crypto Sentinel` и `Gerald Sniper`, которым нужны ресурсы для обработки WebSocket потоков и расчета графиков (`matplotlib / pandas`).
- **Ограничение контекста:** Лимит контекста установлен в 8192 токена, что не позволяет загружать большие куски кода, документации или длинную историю диалога (приходится делать постоянную саммаризацию).
- **Сложность JSON:** Локальная 7B модель иногда ошибается при генерации сложного JSON для вызова инструментов, особенно если контекст перегружен.

## 3. Цель миграции

Основная задача — **заменить локальный LLM-движок на облачные API**, используя лучшие бесплатные модели (Free Tiers), чтобы:

- Снизить Latency (время отклика) с 90 секунд до 2-5 секунд.
- Освободить GPU/CPU сервера для торговых скриптов и вычислительных задач (KDE, базы данных).
- Получить идеальный Tool Calling (Function Calling) и следование JSON-схеме.
- Значительно расширить контекстное окно (до 100k+ токенов).

**Кандидаты на использование (Бесплатные API):**

- **Groq** (Лучший для минимального Latency, Llama-3.3-70B, Mixtral).
- **Google AI Studio (Gemini 1.5 Pro / Flash)** (Огромный контекст 1M/2M токенов, отличный бесплатный тир).
- **NVIDIA Build API** (Доступ к топовым моделям вроде Llama 3.1 405B, Nemotron).
- **OpenRouter** (Доступ к бесплатным моделям: `google/gemini-2.0-flash-exp:free`, `meta-llama/llama-3.3-70b-instruct:free` и др.).
- **GitHub Models** (Доступ к GPT-4o, Claude 3.5 Sonnet, o1-mini для тестов).

## 4. Задача для ИИ-консилиума

Консилиум моделей должен разработать **Техническую спецификацию и пошаговый план внедрения** новой архитектуры, ответив на следующие вопросы:

### Архитектурные вопросы:

1. **Интерфейс маршрутизации (Router):** Как лучше всего реализовать переключение между провайдерами? (Например, использовать библиотеку `LiteLLM` для унификации API-запросов под стандарт OpenAI, или написать кастомный асинхронный роутер на `httpx` / родных SDK). Нужно учесть rate-лимиты бесплатных тарифов (fallback логика: если Groq отдал 429, переключаемся на Gemini/OpenRouter).
2. **Безопасность (Security):** Как гарантировать, что при отправке контекста в облако (особенно при Tool Calling инструмента `read_file` или `execute_python_code`) мы не сольем API-ключи от Bybit, Telegram токены или приватные базы данных? (Необходим гайдлайн/фильтр перед отправкой логов в облако).
3. **Structured Output (JSON):** В текущем коде Pydantic-модель `AgentOutput` конвертируется в JSON-схему и скармливается локальной LLM. Как максимально надежно реализовать возврат `AgentOutput` через зоопарк облачных API (какие модели поддерживают `response_format={ "type": "json_schema" }` строго, а какие требуют System Prompting)?
4. **Гибридный режим (Local Fallback):** Построить логику, при которой: если все облачные провайдеры недоступны (нет интернета, бан IP, исчерпаны лимиты), система должна автоматически откатиться на `LlamaEngine` (локальная Qwen-7B), чтобы "Gerald" никогда не "умирал".

### Требования к коду-результату:

Консилиум должен спроектировать новый класс `CloudLLMEngine`, который реализует тот же интерфейс, что и текущий локальный движок, в частности асинхронный метод:
